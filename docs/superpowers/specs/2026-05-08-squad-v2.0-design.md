## v1.1 — Handoff visualization

### Mechanism

After each agent succeeds, before its dependents start, the orchestrator scans the agent's artifact directory and emits a `HandoffEvent`. No protocol change — purely observability over the existing DAG and artifact-based handoff model.

```ts
type HandoffEvent = {
  from: string,                                // agent name
  to: string[],                                // dependents per the DAG
  files: { path: string, bytes: number }[],    // artifacts produced
}
```

### TUI changes (`src/tui.ts`)

- Two-column layout when terminal width ≥ 100 cols:
  - Left pane (existing): agent status table.
  - Right pane (new): "Handoffs" — scrolling list of the last ~10 handoffs in `A → B: filename (size)` format.
- Single-column fallback below 100 cols: handoffs print inline beneath status (no right pane).
- No new TUI library. Continue using raw ANSI + Chalk per existing convention.
- `--no-tui` mode: events still emitted to JSON output (the existing `--json` payload gets a top-level `handoffs: HandoffEvent[]` array).

### Files

- `src/orchestrator.ts` — emit `HandoffEvent` after each agent success
- `src/tui.ts` — two-pane layout + ring buffer of recent handoffs
- `src/handoff.ts` (new) — `scanArtifacts(dir): { path, bytes }[]`
- `tests/handoff.test.ts` (new) — assert event fires with correct files/dependents; assert `(no artifacts)` rendered when agent produces none; assert no event when agent has no dependents

### Edge cases

- Agent produces no files → still emits with empty `files[]`; TUI renders `A → B: (no artifacts)`
- Agent has no dependents → no event emitted
- Agent fails → no event (failure already covered by status pane)

## v1.2 — Recipe registry

### Hosting

- Public companion repo: `omermaksutii/squad-recipes` (created during release, not part of squad codebase).
- Layout:
  - `index.json` at repo root.
  - `recipes/<name>.json` per recipe.
- Index URL: `https://raw.githubusercontent.com/omermaksutii/squad-recipes/main/index.json`.
- Override via `$SQUAD_REGISTRY_INDEX` env var (for testing + custom registries).

### Index schema

```json
{
  "version": 1,
  "recipes": [
    {
      "name": "auth-flow",
      "description": "Build login + RLS for Supabase apps",
      "author": "@alice",
      "url": "https://raw.githubusercontent.com/omermaksutii/squad-recipes/main/recipes/auth-flow.json",
      "tags": ["supabase", "auth", "rls"],
      "version": "1.0.0"
    }
  ]
}
```

### CLI commands

- `squad search <query>` — fetch index, fuzzy-match `query` against name/description/tags (case-insensitive substring), score: name match > desc match > tag match. Print table: `name  description  author`.
- `squad install <name>` — look up `name` in index, fetch recipe URL, validate JSON, save to `~/.squad/recipes/<name>.json`.
- Existing `squad install <url|file>` keeps working. Disambiguation:
  - Starts with `http://` or `https://` → URL fetch (existing).
  - Contains `/` or ends with `.json` → local file path (existing).
  - Otherwise → registry name lookup (new).
- Name collisions with built-in recipes: error out unless `--force` is passed. Built-ins always win without `--force`.

### Caching

- Cache file: `~/.squad/.registry-cache.json`.
- Stores `{ etag, lastFetched, index }`.
- Refresh policy: if cache age > 24h, refetch with `If-None-Match: <etag>`. On 304, bump `lastFetched` and reuse. On 200, replace.
- Offline: if fetch fails (network, DNS, 5xx) and cache exists → use cache + warn user. If neither → fail with `Registry unreachable; no cache. Try again or set $SQUAD_REGISTRY_INDEX.`

### Files

- `src/registry.ts` (new) — `fetchIndex()`, `searchIndex(query, index)`, `installFromRegistry(name)`
- `src/commands/search.ts` (new) — `squad search` command
- `src/commands/install.ts` (modify) — add registry name path
- `tests/registry.test.ts` (new) — mock fetch via `vi.fn()`; assert search ranking, install path, cache hit, offline fallback, name collision behavior

### Initial state

- `omermaksutii/squad-recipes` ships with `{ "version": 1, "recipes": [] }`. Empty registry. We seed it after public flip with one or two example recipes.

## v1.3 — Docker sandbox

### Activation

- New flag: `--sandbox` (boolean). Default off — backward compat preserved.
- Optional refinement: `--sandbox-network=<bridge|none>` (default `bridge`). Sub-options designed but only network is exposed initially; mem/cpu defaults are fixed.

### Container

- Image: `ghcr.io/omermaksutii/squad-runner:2` (major-version tag tracks the squad major).
- Base: `node:lts-slim`. Installs: `claude` CLI globally + `git` + basic build tools.
- Built by a new workflow `.github/workflows/runner-image.yml` on tag push from the squad repo. Pushes to `ghcr.io/omermaksutii/squad-runner` (public).

### Run-time behavior

Per agent (one container per agent run, lifetime = that agent's claude invocation):

- Mount: `<workspace>:/work` (rw), `cwd=/work`.
- Mount: `.squad/runs/<id>/<agent>/:/work/.squad/runs/<id>/<agent>/` (rw) — already inside workspace, but explicit for clarity.
- Network: `--network=bridge` by default (claude API needs outbound). `--sandbox-network=none` for air-gapped users who run a local proxy.
- Resources: `--memory=2g --cpus=2`.
- Env: pass through `ANTHROPIC_API_KEY` only. No other host env leaks into the container.
- User: on Linux, `--user $(id -u):$(id -g)` so written files match host ownership. No-op on macOS (Docker Desktop handles this).
- Cleanup: `--rm` on every container.

### Adapter interface

Designed so adding E2B later is a 200-line addition, not a refactor.

```ts
// src/sandbox/types.ts
interface SandboxAdapter {
  name: 'docker' | 'e2b'
  preflight(): Promise<void>            // throws if backend unavailable
  runAgent(agent: AgentDef, ctx: RunContext): Promise<RunResult>
}
```

### Files

- `src/sandbox/types.ts` (new)
- `src/sandbox/docker.ts` (new) — implements `SandboxAdapter` for Docker
- `src/runner.ts` (modify) — when `ctx.sandbox` is set, route through adapter instead of direct `spawn('claude', ...)`
- `src/commands/doctor.ts` (modify) — add Docker checks when `--sandbox` flag is observed
- `tests/sandbox-docker.test.ts` (new) — gated behind `RUN_DOCKER_TESTS=1` env var; default test run skips it
- `Dockerfile.runner` (new, repo root) — image definition
- `.github/workflows/runner-image.yml` (new) — build + push to ghcr.io on tag

### Doctor checks (when `--sandbox` is in scope)

- `docker --version` succeeds
- `docker info` succeeds (daemon reachable)
- `docker pull ghcr.io/omermaksutii/squad-runner:2` succeeds (or image already cached)

### Edge cases

- Docker not installed → preflight throws with `Install Docker, or run without --sandbox.`
- Image pull fails (network) → preflight throws with the underlying error + `If offline, pre-pull the image.`
- `ANTHROPIC_API_KEY` not set → same error as today (claude itself errors). No special handling.

## Cross-cutting

### Versioning

- `package.json`: bump to `2.0.0`.
- Major-bump rationale: Docker sandbox is a UX-shaping feature worth signaling; registry adds new commands; handoff TUI changes default rendering. Even though all are additive (no breaking changes), bumping major matches user expectation that "squad 2" is a recognizably new thing.

### Backward compat

- Zero breaking changes. All new behavior is opt-in:
  - `--sandbox` is off by default
  - Two-pane TUI degrades to single-column on narrow terminals (matches today's behavior)
  - `squad search` / `squad install <name>` are new commands; existing `install <url|file>` unchanged
  - Existing recipes, runs, CI configs, JSON output schema all continue to work (the new top-level `handoffs[]` is an addition, not a change to existing fields)

### Release flow (Approach A from brainstorm)

1. Implement v1.1 + v1.2 + v1.3 on `main` (still private)
2. Create public `omermaksutii/squad-recipes` repo, push initial empty index
3. Add Dockerfile.runner + runner-image workflow to squad repo
4. Bump `package.json` to `2.0.0`, write CHANGELOG entry
5. Tag `v2.0.0` — this triggers the runner-image workflow, pushing `ghcr.io/omermaksutii/squad-runner:2`
6. `npm publish` (user is authed as `omermaksutii`)
7. User flips the squad repo public on GitHub
8. (Follow-up, not blocking) v2.1.0: create `squad-action` repo, list on Marketplace

### Testing strategy

- v1.1: unit tests with mocked artifact dirs (`tests/handoff.test.ts`).
- v1.2: unit tests with mocked `fetch` (`tests/registry.test.ts`). No network in CI.
- v1.3: tests gated behind `RUN_DOCKER_TESTS=1`. CI default skips them. Local-run instructions in CHANGELOG.
- Existing tests must continue to pass unchanged. `vitest run` is the gate.

### Documentation

- README gets new sections under "Recipes" (registry usage), "TUI" (handoffs), and "Running" (sandbox flag).
- CHANGELOG: 2.0.0 entry covering all three.
- No CLAUDE.md needed — this is the squad project itself.
